# Phase 3 — Manifest Driven Media UI

## Overview
Switch the browser from client-side media discovery/hardcoded station copy to the backend manifest, and remove eager MP4/FBX startup loading. Preserve visible fallback when media API or assets fail.

## Requirements
- Browser bootstrap fetches `/api/scene/{sceneId}` and `/api/tour/{tourId}`, then `/api/media/{sceneId}`; use the media manifest as production source for model/video metadata.
- Remove production dependency on `import.meta.glob('/making_step/*.mp4')` at `apps/web/src/components/ExhibitionWall/ExhibitionWall.js:80-86`.
- Remove production dependency on `PAPERMAKING_STEPS` hardcoded station narrations at `apps/web/src/components/ExhibitionWall/ExhibitionWall.js:4-75`; approved copy comes from manifest.
- Remove startup video preload loops at `apps/web/src/main.js:198-201,598-601`; keep `VideoDisplay` lazy semantics from `apps/web/src/components/VideoDisplay/VideoDisplay.js:53-89`.
- Replace the startup all-FBX gate at `apps/web/src/main.js:180-191` with a lazy model registry; base scene/fallback should render before all model files finish.
- Keep current FBX files and current placement logic where practical; do not convert to GLB in this pass.
- If manifest fetch fails, media degrades only; do not hardcode autogenerated fallback descriptions.

## Data flow
1. Browser bootstrap fetches `/api/scene/{sceneId}`, `/api/tour/{tourId}`, and then `/api/media/{sceneId}` through `apps/web/src/media/client.js`.
2. `manifest-adapter` indexes `assets[]` by `assetId`, maps each `processStations[]` entry to `{stepNum,title,narration,videoUrl}` using its referenced video asset, and maps model roles to existing placement/lazy loader hooks.
3. `createExhibitionWall(scene, mediaStations)` builds station meshes from manifest data; missing video yields mock canvas, but missing copy does not invent educational text.
4. `model-registry` creates lazy loader functions; a model asset loads on first activation/visibility/tour need, then caches success/failure per asset.
5. `VideoActivationSystem` calls `VideoDisplay.play()` near active distance, so `VideoDisplay.load()` is still first real media request.
6. On manifest/API error, adapter returns degraded media state; tour rail and QA/TTS UI still use their existing scene/tour APIs, and the approved 5-step tour remains exactly 5 steps.

## Implement
1. Add manifest fetch client with one retry-free request and bounded timeout; no client cache abstraction beyond a simple per-page promise.
2. Add adapter that validates minimal manifest fields client-side and falls back to degraded state if missing/malformed.
3. Rewrite `createExhibitionWall` to take explicit station data; remove `import.meta.glob`, `detectedVideos`, and autogenerated additional-step copy from the production path.
4. Change `ExhibitionStation` constructor to consume station view model fields (`order`, `title`, optional `titleEn`, narration, `videoUrl`) while preserving mock video support.
5. Split `main.js` startup: create renderer/corridor/wall/fallback shell first, fetch scene/tour/media, then load models lazily by asset role. Do not wait on all FBX in one `Promise.all` before removing loading screen.
6. Keep existing geometric fallback block from `apps/web/src/main.js:496-609`, but route it through manifest-aware station/model fallback where possible.
7. Keep avatar static preview exports backwards-compatible; internally prefer manifest asset metadata when a media manifest is available.
